🏥 DOCTORPRO — FINAL ENCOUNTER UX DESIGN FLOW

⸻

1️⃣ Core Principle
	•	One Visit = One Encounter
	•	Navigation ≠ State Change
	•	Only explicit buttons change state
	•	No automatic encounter creation
	•	No state rollback based on data

Single Source of Truth:

Encounter.status


⸻

2️⃣ Encounter Status Lifecycle

CREATED
↓
PRE_CONSULTATION_IN_PROGRESS
↓
PRE_CONSULTATION_COMPLETED
↓
CONSULTATION_IN_PROGRESS
↓
CONSULTATION_COMPLETED
↓
CLOSED (optional future)

Optional terminal:

CANCELLED

State only changes by button action.

⸻

3️⃣ Entry Logic (When Doctor Selects Patient)

System checks:

If Active Encounter Exists:

(status in CREATED / PRE_CONSULTATION_IN_PROGRESS / CONSULTATION_IN_PROGRESS)

👉 Auto-resume that encounter
No popup.

If Last Encounter Completed:

👉 Show:
	•	Start New Visit button

If No Encounter:

👉 Create new encounter → Redirect to Pre

⸻

4️⃣ Pre-Consultation Page UX

Allowed When:

status = CREATED or PRE_CONSULTATION_IN_PROGRESS

Buttons:
	•	Save Draft
	•	Complete & Start Consultation
	•	Start New Visit (secondary)

⸻

On “Complete & Start Consultation”

status = PRE_CONSULTATION_COMPLETED
→ redirect to Consultation
→ set status = CONSULTATION_IN_PROGRESS


⸻

If Consultation Already Started

Pre becomes:
	•	Read-Only
	•	Show banner:
“Consultation already started. Pre locked.”

No editing allowed.

⸻

5️⃣ Consultation Page UX

Allowed When:

status = CONSULTATION_IN_PROGRESS

Buttons:
	•	Save Draft
	•	End Consultation
	•	Start New Visit
	•	View Pre (popup)

⸻

6️⃣ End Consultation Logic

On click:

Popup:

Are you sure you want to end this consultation?
This will lock the encounter.

If confirmed:

status = CONSULTATION_COMPLETED
consultation.completed_at = now()

Then:
	•	Lock encounter
	•	Disable editing
	•	Redirect to:
	•	Smart Queue OR
	•	Patient Dashboard

⸻

7️⃣ Start New Visit Logic

Visible in:
	•	Pre page
	•	Consultation page
	•	Completed encounter

⸻

Case A — During Active Visit

Popup:

This visit is still active.
End this visit and start a new one?

If confirmed:
	•	Mark current as CONSULTATION_COMPLETED (or CANCELLED if pre only)
	•	Create new Encounter
	•	Redirect to Pre of new encounter

⸻

Case B — After Completed Visit

No popup required.
	•	Create new Encounter
	•	Redirect to Pre

⸻

8️⃣ Back Button Behavior

Browser Back:

❌ Does NOT change state
❌ Does NOT create new encounter
❌ Does NOT rollback

On page load:

System reads:

Encounter.status

Renders accordingly.

Always deterministic.

⸻

9️⃣ Pre View During Consultation

Instead of navigating back:

Consultation page shows:

“View Pre Details” button

Opens right-side popup.

Read-only.

Prevents navigation confusion.

⸻

🔟 Locking Rules Table

Status	Pre	Consultation
CREATED	Editable	Hidden
PRE_CONSULTATION_IN_PROGRESS	Editable	Hidden
PRE_CONSULTATION_COMPLETED	Editable (until consult starts)	Hidden
CONSULTATION_IN_PROGRESS	Read-only	Editable
CONSULTATION_COMPLETED	Read-only	Read-only
CANCELLED	Read-only	Read-only


⸻

1️⃣1️⃣ What NEVER Happens

❌ No automatic new encounter creation
❌ No state change from navigation
❌ No rollback if form incomplete
❌ No popup asking resume/new for active encounter
❌ No duplicate consultation inside same encounter

⸻

1️⃣2️⃣ Doctor Mental Model (Validated)

Doctor expects:
	•	If I leave and come back → my work is there.
	•	If I end → visit is closed.
	•	If I want new → I click Start New Visit.
	•	System should not ask confusing branching questions.

This UX satisfies all.

⸻

1️⃣3️⃣ Technical Enforcement Rules

Backend must:
	•	Validate status before allowing edit
	•	Reject updates if locked (403)
	•	Never infer state from data presence
	•	Always change status explicitly via service layer

⸻

1️⃣4️⃣ Why This Architecture Is Stable

✔ No UX ambiguity
✔ No duplicate records
✔ Audit-safe
✔ Billing-safe
✔ Scalable to enterprise
✔ Ready for future:
	•	Rx preview
	•	Billing
	•	Insurance
	•	Follow-up visits
	•	Teleconsultation

⸻
