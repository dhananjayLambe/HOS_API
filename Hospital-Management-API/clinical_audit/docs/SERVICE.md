# Clinical Audit Service (M3.2)

Centralized, fail-open write path for immutable clinical audit records.

Business modules must call **`ClinicalAuditService.record()`** only. They must never create `ClinicalAudit` rows directly.

**New to this API?** See [HOW_TO_USE.md](HOW_TO_USE.md) for copy-paste examples, required fields, correlation IDs, and where to call from in your services.

## Architecture

```
ClinicalAuditService.record()
        │
        ▼
AuditRequestValidator  →  ValidatedAuditRequest
        │
        ▼
ClinicalAuditBuilder   →  unsaved ClinicalAudit
        │
        ▼
ClinicalAuditRepository →  PostgreSQL (insert-only)
```

## Public API

```python
from clinical_audit.services import ClinicalAuditService, AuditRecordResult
from clinical_audit.enums import AuditAction, AuditSource, ClinicalEntity

result: AuditRecordResult = ClinicalAuditService.record(
    action=AuditAction.CONSULTATION_STARTED,
    event="Consultation started",
    resource_type=ClinicalEntity.CONSULTATION,
    resource_id=str(consultation_id),
    source=AuditSource.DOCTOR,
    user_id=str(user_id),
    organization_id=str(clinic_id),
    patient_account_id=str(patient_account_id),
    consultation_id=str(consultation_id),
    encounter_id=str(encounter_id),
    payload={"status": "started"},
    correlation_id=correlation_id,  # optional; inherits from LogContext
)
```

### Result object

| Field | Description |
|---|---|
| `success` | `True` when the audit row was persisted |
| `audit_id` | UUID of the saved record (when successful) |
| `correlation_id` | Workflow correlation ID used for the attempt |
| `error` | Human-readable failure message |
| `error_type` | Exception class name (e.g. `AuditValidationError`) |

## Failure isolation

Audit failures **must not** interrupt patient-care workflows.

- By default, `record()` catches all `ClinicalAuditError` subclasses, logs a warning with correlation context, and returns `success=False`.
- Pass `raise_on_failure=True` only in tests or admin tooling when you need exceptions to propagate.

## Correlation integration

Resolution order for `correlation_id`:

1. Explicit argument to `record()`
2. Active `LogContext` from Phase 2 (`shared.logging.ContextManager`)
3. New UUID4 generated by the builder

`request_id`, `user_id`, `user_role`, and patient/consultation/encounter IDs are similarly backfilled from `LogContext` when omitted.

## API-to-model mapping

| Service parameter | `ClinicalAudit` field |
|---|---|
| `action` | `action` |
| `event` | `event` |
| `module` | `module` (derived from action prefix if omitted) |
| `source` | `source` |
| `user_id` / `user_role` | `user_id` / `user_role` |
| `organization_id` | `new_value._meta.organization_id` |
| `payload` | `new_value.payload` |
| `snapshot` | `previous_value` |
| `correlation_id` | `correlation_id` |
| `request_id` (auto) | `new_value._meta.request_id` |

Extended metadata (`occurred_at`, `timezone`, `application_version`, `service_name`, `hostname`) is stored under `new_value._meta`. See [MODELS.md](MODELS.md) for the full envelope spec.

## Validation

`AuditRequestValidator` enforces:

- Required fields: action, event, resource_type, resource_id, source, user_id, organization_id
- Enum membership (`AuditAction`, `AuditSource`, `ClinicalEntity`)
- UUID format for `organization_id` and optional `correlation_id`
- JSON-serializable `payload` and `snapshot` within size limits
- Optional reference existence checks against `Clinic`, `PatientAccount`, `PatientProfile`, `Consultation`, `ClinicalEncounter`

Disable reference checks in tests or batch imports with `validate_references=False`.

## Repository (read path)

`ClinicalAuditRepository` exposes append-only writes and read helpers:

- `save`, `bulk_save`
- `get_by_event_id`, `get_by_correlation_id`
- `filter_by_resource`, `filter_by_patient`

Query/timeline APIs for external consumers are planned in M3.6.
