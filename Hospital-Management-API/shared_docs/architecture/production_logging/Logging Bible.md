Document Type: Governing Architecture Specification

Version: 1.0

Status: Production Ready

Owner: DoctorProCare Platform Team

⸻

1. Purpose

This document defines the Observability, Logging, Clinical Audit, Monitoring, and Production Support Architecture for the DoctorProCare platform.

It is the single source of truth governing how every important event generated by the platform is recorded, monitored, audited, searched, and investigated.

This document applies to every backend service, scheduled task, API, and future microservice.

If any implementation differs from this document, this document takes precedence.

⸻

2. Goals

The primary objective of this architecture is to ensure that every production issue can be investigated quickly while maintaining a complete clinical audit trail for EMR operations.

The platform must answer the following questions within a few minutes.

Technical

* Why did the API fail?
* Why did the Recommendation Engine fail?
* Which Celery task failed?
* Why was the WhatsApp message not delivered?
* Why did report upload fail?
* Why was a booking rejected?

Business

* Was the recommendation generated?
* Was the booking submitted?
* Which laboratory accepted the booking?
* Which report was uploaded?
* Was the report delivered?

Clinical

* Which doctor viewed the patient record?
* Who generated the prescription?
* Who modified the prescription?
* Who uploaded the report?
* Who downloaded the report?
* When was the consultation completed?

3. Design Principles

Principle 1

Every critical workflow must be observable.

⸻

Principle 2

Clinical Audit and Application Logging are different.

Never mix them.

⸻

Principle 3

Business Audit and Technical Logs are different.

Never mix them.

⸻

Principle 4

Every workflow must have a Correlation ID.

⸻

Principle 5

Business events must be permanently auditable.

⸻

Principle 6

Logging must never expose sensitive information unnecessarily.

⸻

Principle 7

Logging should be lightweight today and enterprise-ready tomorrow.

4. Observability Architecture

DoctorProCare consists of four independent layers.

⸻

Layer 1 — Application Logs

Purpose

Technical troubleshooting.

Storage

CloudWatch Logs (Production)

Console Logger (Development)

Retention

30–90 Days

Examples

* API Requests
* API Responses
* Celery Tasks
* Recommendation Engine
* Booking Engine
* WhatsApp Integration
* S3 Upload
* Redis
* Database Queries
* Performance Metrics
* Exceptions

Application logs are temporary operational records.

⸻

Layer 2 — Clinical Audit Trail

Purpose

Permanent healthcare audit history.

Required for EMR accountability.

Storage

PostgreSQL

Retention

Permanent

Clinical Audit records must never be modified or deleted.

Examples

* Patient Record Viewed
* Consultation Started
* Consultation Completed
* Diagnosis Added
* Diagnosis Updated
* Prescription Generated
* Prescription Updated
* Prescription Downloaded
* Investigation Added
* Investigation Removed
* Report Uploaded
* Report Viewed
* Report Downloaded
* Digital Signature Applied

⸻

Layer 3 — Business Audit

Purpose

Permanent business workflow history.

Storage

PostgreSQL

Retention

Permanent

Examples

* Recommendation Generated
* Recommendation Sent
* Booking Submitted
* Booking Confirmed
* Laboratory Accepted
* Laboratory Rejected
* Laboratory Reassigned
* Report Delivered
* WhatsApp Delivered
* Payment Completed
* Refund Processed

⸻

Layer 4 — Support Trace Index

Purpose

Fast production troubleshooting.

Storage

PostgreSQL

Retention

Permanent

One searchable record should exist for every consultation and major workflow.

The Support Trace Index bridges business searches to technical logs.

Search should be possible using:

* Patient Mobile Number
* Doctor Mobile Number
* Consultation ID
* Encounter ID
* Patient Account ID
* Booking ID
* Diagnostic Order ID
* Report ID
* WhatsApp Message ID
* Recommendation ID
* Date Range

The result should return:

* Correlation ID
* Consultation
* Encounter
* Booking
* Recommendation
* Current Status

Support engineers should search this table first before searching CloudWatch.

* Service

10. Clinical Audit Requirements

Every clinical action affecting patient care must generate an audit record.

Each audit record should include:

* Event Timestamp
* Event Type
* User ID
* User Role
* Patient Account ID
* Patient Profile ID
* Consultation ID
* Encounter ID
* Source Module
* Action
* Previous Value (where applicable)
* New Value (where applicable)
* Correlation ID
* IP Address (optional)
* Device Information (optional)

Audit records are immutable.

↓

↓

14. Alerting

Critical alerts only.

Examples

* API 500 spike
* Celery worker unavailable
* WhatsApp delivery failures
* Report upload failures
* Database unavailable
* Redis unavailable
* Storage failures

Alerts should notify platform administrators by email.

⸻

15. Security & Privacy

Application logs must never contain:

* Passwords
* JWT Tokens
* Access Tokens
* Refresh Tokens
* OTPs
* Clinical report contents
* Prescription PDF contents
* Payment secrets

Avoid storing raw patient phone numbers in CloudWatch logs.

Use Patient Account ID or Correlation ID where possible.

Patient and doctor searches should be performed using the Support Trace Index.

18. Success Criteria

The logging platform is considered production-ready when:

* Every critical workflow produces structured application logs.
* Every consultation generates a Correlation ID.
* Clinical audit events are permanently stored.
* Business workflow events are permanently stored.
* Support can locate any patient journey using business identifiers.
* Developers can diagnose production issues using CloudWatch logs.
* The architecture supports future enterprise observability without redesign.
* Logging remains lightweight, maintainable, and suitable for a growing EMR platform.
