Purpose

This document defines the Correlation Framework used throughout the DoctorProCare platform.

The Correlation Framework enables every business workflow to be traced end-to-end across APIs, Celery workers, WhatsApp notifications, laboratory operations, report uploads, and future services.

Its purpose is to ensure that any production issue can be reconstructed using a single Correlation ID.

⸻

Objectives

The Correlation Framework must:

* Track an entire patient workflow.
* Link all related technical logs.
* Connect application logs with audit events.
* Simplify production troubleshooting.
* Support distributed processing.
* Scale to future microservices.

⸻

What is a Correlation ID?

A Correlation ID is a unique identifier assigned to a workflow when it begins.

Every subsequent operation generated by that workflow reuses the same Correlation ID.

The Correlation ID never changes during the lifetime of the workflow.

⸻

A single patient consultation may generate many downstream operations.

All of them share the same Correlation ID.

⸻

Correlation ID Generation

A Correlation ID is created only once.

Generation occurs at the first entry point into the workflow.

Examples:

* Incoming REST API request
* WhatsApp Flow submission
* Scheduled job
* Administrative action
* Future mobile application request

If an incoming request already contains a Correlation ID, it should be reused after validation.

⸻

Propagation Rules

Once generated, the Correlation ID must be propagated automatically to every downstream component.

Propagation includes:

* Django request context
* Service layer
* Repository layer
* Celery tasks
* WhatsApp services
* Recommendation engine
* Laboratory workflow
* Report services
* Storage operations
* Audit services

Developers should never manually copy Correlation IDs between layers.

⸻

Standard Correlation Context

The runtime context should contain:

* Correlation ID
* Request ID
* User ID
* User Role
* Patient Account ID
* Patient Profile ID
* Consultation ID
* Encounter ID
* Recommendation ID
* Booking ID
* Laboratory ID
* Report ID
* WhatsApp Message ID

Not every workflow will populate every field.

The Correlation ID is the only mandatory field.

⸻

Relationship Between IDs

DoctorProCare contains several business identifiers.

Correlation ID
        │
        ├── Request ID
        ├── Consultation ID
        ├── Encounter ID
        ├── Recommendation ID
        ├── Booking ID
        ├── Laboratory ID
        ├── Report ID
        └── WhatsApp Message ID

The Correlation ID links them together.

⸻

Middleware Responsibilities

The request middleware should:

* Generate a Correlation ID if absent.
* Reuse an existing Correlation ID when appropriate.
* Create a Request ID.
* Store both values in the request context.
* Make them available throughout the request lifecycle.

Business code should never create Correlation IDs.

⸻

Celery Propagation

Every Celery task must receive the Correlation ID.

The worker must restore the context before executing business logic.

Example workflow:

API Request
↓
Recommendation Service
↓
Celery Task
↓
WhatsApp Send Task
↓
CloudWatch Log
↓
Same Correlation ID

This enables tracing across asynchronous processing.

⸻

Logging Requirements

Every application log must include:

* Correlation ID
* Request ID
* Module
* Action
* Timestamp

Developers should never manually append Correlation IDs.

The Logger Framework should inject them automatically.

⸻

Clinical Audit

Clinical Audit records should reference:

* Correlation ID
* Consultation ID
* Encounter ID
* Patient Account ID

This enables linking audit records with technical logs.

⸻

Business Audit

Business Audit events should reference:

* Correlation ID
* Recommendation ID
* Booking ID
* Report ID
* WhatsApp Message ID

Business history becomes searchable through the Correlation Framework.

⸻

Support Trace Integration

Support engineers should not search CloudWatch directly.

Investigation workflow:

Patient Mobile
↓
Support Trace Index
↓
Consultation
↓
Correlation ID
↓
Clinical Audit
↓
Business Audit
↓
CloudWatch Logs
↓
Root Cause

The Correlation Framework connects every stage of this investigation.

⸻

Error Handling

If a Correlation ID cannot be resolved:

* Generate a new Correlation ID.
* Record a warning.
* Continue processing.
* Never fail a business workflow because of missing correlation metadata.

Observability must never interrupt patient care.

⸻

Security

Correlation IDs must not contain:

* Patient names
* Mobile numbers
* Email addresses
* Medical information
* Booking numbers
* Sequential identifiers

Use randomly generated UUIDs.

Correlation IDs are technical identifiers only.

⸻

Performance

The Correlation Framework should:

* Generate IDs only once.
* Store context efficiently.
* Avoid repeated lookups.
* Support concurrent requests.
* Support asynchronous execution.
* Have negligible runtime overhead.

⸻

Future Compatibility

The Correlation Framework is designed to support:

* Microservices
* Event-driven architecture
* OpenTelemetry
* Distributed tracing
* Service mesh environments
* External partner integrations

No redesign should be required.

⸻

Acceptance Criteria

The Correlation Framework is complete when:

* Every workflow receives a Correlation ID.
* The Correlation ID is automatically propagated.
* Application logs include the Correlation ID.
* Clinical Audit references the Correlation ID.
* Business Audit references the Correlation ID.
* Celery tasks preserve the Correlation ID.
* Support can reconstruct an end-to-end patient journey using a single Correlation ID.
* Missing Correlation IDs never interrupt business processing.

⸻

Summary

The Correlation Framework is the backbone of DoctorProCare’s production observability strategy.

By assigning a single Correlation ID to every workflow and propagating it across APIs, background tasks, audits, and integrations, the platform provides complete end-to-end traceability for patient journeys, simplifies production troubleshooting, and establishes a strong foundation for future distributed systems and enterprise observability.
