DoctorPro Diagnostic Catalog Architecture (Phase 1)

Purpose

This document defines the production-grade diagnostic catalog architecture for DoctorPro.

The catalog becomes the foundation for:

* Consultation investigations
* Diagnostic orders
* Routing engine
* Lab marketplace
* Pricing
* Search
* Recommendations
* Packages
* AI recommendation systems
* Analytics
* Reporting

This architecture is designed for:

* Long-term scalability
* Safe imports
* Idempotent synchronization
* Stable business identifiers
* Production-safe healthcare catalog management

Services Architecture

Services should ALWAYS map to subcategories.

Correct:

PATH
    ↓
HEM
    ↓
LAB-CBC

Incorrect:

LAB-CBC → PATH

Reason:

* Better analytics
* Better routing
* Better recommendations
* Better filtering
* Better search
* Better package organization

⸻

services/*.csv Format

Recommended format:

ordering,code,name,category_code,short_name,sample_type,home_collection_possible,tat_hours_default,is_active

Example:

10,LAB-CBC,Complete Blood Count,HEM,CBC,Blood,true,24,true
20,LAB-ESR,Erythrocyte Sedimentation Rate,HEM,ESR,Blood,true,24,true
30,LAB-HBA1C,HbA1c,DIABETIC,HbA1c,Blood,true,24,true

⸻

Service Code Rules

Service codes are immutable business identifiers.

Examples:

LAB-CBC
LAB-HBA1C
LAB-CRP
RAD-XRAY-CHEST
RAD-USG-ABDOMEN
CARDIO-ECG

Never rename codes after production rollout.

These codes become dependencies for:

* Pricing
* Routing
* Orders
* Packages
* Analytics
* Recommendations
* Reports

CSV Import Strategy

Recommended Import Command

Single orchestration command:

python manage.py sync_diagnostic_catalog

This command should:

1. Import categories
2. Import subcategories
3. Import all services
4. Import packages
5. Import package items

⸻

Import Order

categories.csv
    ↓
subcategories.csv
    ↓
services/*.csv
    ↓
packages/*.csv
    ↓
package_items/*.csv

⸻

Import Philosophy

Use update_or_create

Always use:

update_or_create()

Never use:

* blind inserts
* delete + recreate
* hard resets in production

Reason:

* preserves UUIDs
* preserves pricing mappings
* preserves routing references
* preserves historical orders

⸻

Soft Delete Strategy

Never automatically hard-delete catalog rows.

Instead:

is_active = False

Reason:

* historical orders exist
* reports exist
* pricing references exist
* routing references exist

⸻

Recommended Import Features

Future-safe importer capabilities:

* Dry run mode
* Validation summary
* Duplicate detection
* Missing category detection
* Invalid hierarchy detection
* CSV schema validation
* Import statistics

Example:

python manage.py sync_diagnostic_catalog --dry-run

⸻

Validation Rules

Categories

* unique code
* valid ordering
* no circular hierarchy

Subcategories

* parent_code must exist
* no self-parenting
* unique code

Services

* category_code must exist
* immutable service code
* no duplicate semantic services

⸻

Important Operational Rules

1. Services Should Be Canonical

Do NOT create:

CBC
COMPLETE-BLOOD-COUNT
HEMOGRAM

as separate services.

Instead:

LAB-CBC

with:

* synonyms
* tags
* aliases

⸻

2. Root Categories Should Rarely Change

Changing root categories later impacts:

* analytics
* dashboards
* recommendations
* routing

⸻

3. Service Codes Must Never Change

Changing service codes later breaks:

* routing
* pricing
* packages
* orders
* integrations
